# Contributing a New Artifact to Sigil

This guide covers the full lifecycle: scaffold → fill → validate → submit a PR.

---

## 1. Understand the kinds

Pick the kind that matches the intent:

| Kind       | When to use                                  | Keyword signal          |
| ---------- | -------------------------------------------- | ----------------------- |
| **skill**  | Step-by-step AI guidance invoked on demand   | "When I want to…"       |
| **agent**  | Persistent AI persona with an ongoing role   | "An AI that always…"    |
| **rule**   | Always-on coding conventions (style, naming) | "Always follow…"        |
| **prompt** | Parameterised one-shot command               | "A command to quickly…" |

If unsure, use a **skill** — it is the most flexible kind.

---

## 2. Understand platform targeting (the DRY rule)

By default, every artifact propagates to **all AIs that support its kind** — that is the DRY default.
Right now that means Claude Code and GitHub Copilot (both support all four kinds).

You can restrict an artifact to a specific AI at creation time:

```bash
sigil new skill --name ef-core --language csharp --platforms claude --yes
```

You can widen or narrow it later:

```bash
sigil retarget csharp/ef-core --add copilot   # widen to both
sigil retarget csharp/ef-core --remove claude  # restrict back
sigil retarget csharp/ef-core --to all         # reset to DRY default
```

---

## 3. Scaffold the file

The interactive wizard handles everything interactively:

```bash
sigil new
```

Or scripted (non-interactive, good for CI/templates):

```bash
# Language-specific skill (the most common case):
sigil new skill --name ef-core-migrations --language csharp --yes

# Shared agent (all languages, all AIs):
sigil new agent --name sql-reviewer --yes

# Claude-only prompt:
sigil new prompt --name explain-diff --platforms claude --yes
```

This creates a file with a schema-derived frontmatter header and a placeholder body under `catalog/`.

For full copy-paste walkthroughs (new skill / new rule / new language end-to-end) →
[docs/guides/authoring.md](docs/guides/authoring.md).

---

## 4. Fill in the body

Open the generated file and fill in the title, description, and body content.

**The most important rule: write the body exactly once, in neutral language.**
Never add "For Claude only:" or "For Copilot:" sections — that defeats the DRY architecture.
The adapters translate format and vocabulary automatically.

### Skill body structure

1. What this skill covers (one sentence)
2. When to invoke it
3. Ordered steps / conventions / patterns

### Rule body structure

Bullet points:

```markdown
- **Convention name.** Explanation. Example if helpful.
```

### Agent body structure

- Persona definition (who you are, what you know)
- When to be invoked
- Behaviour rules and constraints

### Prompt body structure

The instruction text, with `{{placeholder}}` for arguments:

```
Explain the following diff, focusing on {{aspect}}:
```

### DRY inheritance (rules)

If your rule extends an existing rule, set `extends:` in the frontmatter — do not copy the parent's text:

```yaml
extends:
  - shared/clean-code
```

The compiler inlines the parent body at build time.

### Dependency references (skills)

If your skill uses an existing rule or agent, declare it in `uses:` — do not inline their text:

```yaml
uses:
  rules:
    - csharp/dotnet-style
  agents:
    - shared/code-reviewer
```

The compiler resolves and inlines them at build time.

---

## 5. Validate

Run the source validator on your file:

```bash
sigil check catalog/languages/csharp/skills/ef-core-migrations/SKILL.md
```

Then run the full catalog validation (CI gate):

```bash
npm run validate
```

Fix all violations before proceeding. Common violation categories and fixes →
[docs/reference/troubleshooting.md](docs/reference/troubleshooting.md#sigil-validate-reports-schema-violations).

---

## 6. Build and inspect output

Compile to both targets and confirm the output looks right:

```bash
npm run catalog:build
```

Inspect:

- `dist/claude/` — Claude Code plugin layout (SKILL.md files with inlined rules, marketplace.json)
- `dist/copilot/` — GitHub Copilot layout (.github/instructions/, .github/prompts/, AGENTS.md)

---

## 7. Submit a pull request

### Branch naming

```
artifact/<kind>-<name>
# Examples:
artifact/skill-ef-core-migrations
artifact/agent-sql-reviewer
artifact/rule-csharp-async-patterns
```

### What goes in the PR

- The new catalog source file(s) under `catalog/`
- Nothing from `dist/` (the build output is gitignored and regenerated by CI)
- Nothing from `dist-cli/` (compiled TypeScript, also gitignored)

### What reviewers check

- `sigil check <file>` passes (0 violations)
- `npm run validate` passes (0 errors)
- Body is neutral language — no AI-specific sections
- `platforms:` field is intentional (omit for DRY default, or justify the restriction)
- IDs match the path convention (`<language>/<name>` or `shared/<name>`)
- DRY: `extends:` / `uses:` are used rather than copying content from existing artifacts

### PR description template

The repo provides a PR template at `.github/PULL_REQUEST_TEMPLATE.md` — fill in the checklist
when you open the PR.
